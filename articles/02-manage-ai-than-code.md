# 管好AI比写好代码更重要

> Vibe Coding听起来很随意——不看代码，对着AI说话，Accept All。但真正试过的人发现：这需要的能力，和写代码完全是两回事。

---

## 一、你以为在编程，其实在带团队

去年有个词特别火：**Vibe Coding**。创始人说得很玄乎——不看代码，对着AI说话，Accept All，有错粘回去修，能跑就行。

很多人试完发现一个尴尬的事实：**这玩意儿需要的能力，和写代码完全是两回事。**

为什么？因为当你放弃写代码，你就不是程序员了——你是管理者。

你管理的不是人，是AI agents。但管理的本质没变：
- 拆解任务
- 分配角色
- 设定标准
- 验收产出
- 处理异常

听起来是不是很像Tech Lead做的事情？没错，AI时代，每个程序员都要成为自己的Tech Lead。

---

## 二、不看代码看什么？

传统编程默认代码应该一次写对，核心是对过程的控制感。

Vibe Coding把这个假设撕了。你不看代码，当然不可能对过程有信心。那怎么办？

**看结果。**

### 从过程控制到结果导向

| 传统编程 | Vibe Coding |
|---------|-------------|
| 控制每一行代码怎么写 | 控制最终产出是什么样 |
| 过程正确=结果正确 | 结果正确=过程可接受 |
| 写代码的时间占80% | 判断和验收的时间占80% |

Agent说这功能实现不了？别急着翻代码。开个新session，重新描述需求试试。能做出来就继续，做不出来再换。

你追求的是**结果不断逼近正确**，不是一次写对。

### 迭代逼近 vs 一次写对

传统程序员追求"一次写对"，因为改代码成本高。

Vibe Coding追求"快速迭代"，因为：
- 让AI重写比人工改更快
- 不同的描述方式可能得到完全不同的结果
- 试错成本极低，判断成本才是关键

关键是：**你要能在3分钟内判断结果对不对。**

## 三、最大瓶颈：上下文切换

三个agent session同时跑，你是唯一的人类。

Agent跑完只要几分钟，你判断结果需要多久？

犹豫的每一分钟，整个系统都在空转。好的leader三分钟给结论，差的leader犹豫一天。

### 你跟不上，就变成瓶颈

这不是AI能力的问题，是**你的判断速度**的问题。

要提升判断速度，需要：
1. **清晰的验收标准**：什么是"对了"？提前定义
2. **结构化的检查流程**：按清单检查，不靠直觉
3. **快速试错的心态**：错了就kill，不要恋战

### 一个反直觉的做法

一个session跑久了，每次修复都引入新问题，越陷越深。

这时候应该：**关掉这个session，开个新的。**

为什么？上下文被污染了。拖着满是错误尝试的长上下文，模型会变笨。让当前agent把经验沉淀到文档，然后clear session，换全新的继续。

就像团队里一个人在错误方向钻太久，你需要替他做那个他自己做不了的决定：**停下来。**

## 四、最贵的不是订阅费

Claude Code Max一个月两三百美金。很多人觉得贵。

但你算一笔账：
- 弱模型出现幻觉，你花半小时排查纠偏
- 上下文管理不当，agent反复出错
- 需求描述不清，生成十遍都不对

**你的时间才是整个系统里最贵的资源。**

贵的那个一次就对了。省下的时间价值远超订阅费。

### 成本公式

```
总成本 = AI订阅费 + 你的时间成本 + 试错成本

优化目标：最小化（你的时间成本 + 试错成本）
而不是：最小化订阅费
```

当你纠结要不要开Pro版的时候，想想你的时间值多少钱。

## 五、新能力模型

放弃过程掌控不代表放弃质量。质量保障方式变了：**从过程审查变成机制设计。**

你需要的新能力：

### 1. 品味：知道什么是好结果

AI能给你十种实现，你得能一眼看出哪个最好。

这需要：
- 对好代码的认知
- 对架构的理解
- 对业务的洞察

**品味来自经验，AI学不来。**

### 2. 判断力：何时质疑，何时kill

不是每次产出都要质疑，也不是每次都要接受。

判断框架：
- **核心逻辑**：必须质疑，仔细审查
- **边缘功能**：抽查验证
- **样板代码**：可以Accept All

什么时候kill session？
- 连续3次修复都引入新问题
- 上下文已经包含大量错误尝试
- Agent开始"胡说八道"（幻觉）

### 3. 系统思维：设计自运转机制

不要每次都从零开始。设计可复用的机制：

```
┌─────────────────────────────────────────┐
│          质量保障机制设计               │
├─────────────────────────────────────────┤
│                                          │
│  输入标准化                              │
│  ├── PRD模板                            │
│  ├── Prompt模板                         │
│  └── 验收清单                           │
│                                          │
│  过程可追溯                              │
│  ├── ADR记录                            │
│  ├── 变更日志                           │
│  └── 决策理由                           │
│                                          │
│  输出可验证                              │
│  ├── 自动化测试                         │
│  ├── 代码审查                           │
│  └── 集成验证                           │
│                                          │
└─────────────────────────────────────────┘
```

### 4. 表达力：需求描述决定产出质量

这是最被低估的能力。

同样的需求，不同的描述，AI产出天差地别：

| 坏描述 | 好描述 |
|--------|--------|
| 做个登录功能 | 实现用户登录功能，支持邮箱+密码和手机验证码两种方式，登录失败3次锁定15分钟，需要记住登录状态7天 |
| 优化性能 | 首页加载时间从3秒降到1秒以内，主要优化数据库查询和静态资源加载 |
| 加个缓存 | 对热点数据加Redis缓存，TTL设置为1小时，缓存命中率达到80%以上 |

**好描述的三要素**：
1. **做什么**：明确的功能定义
2. **为什么**：上下文和目标
3. **怎么衡量**：可验证的标准

## 六、从传统程序员到AI时代管理者

### 能力对比

| 能力维度 | 传统程序员 | AI时代管理者 |
|---------|-----------|-------------|
| 核心技能 | 写代码 | 管代码生成过程 |
| 质量保障 | 代码审查 | 机制设计 |
| 效率瓶颈 | 手速 | 判断速度 |
| 知识管理 | 自己记住 | 文档化沉淀 |
| 团队协作 | 单兵作战 | 多Agent协作 |
| 学习方向 | 新语言/框架 | Prompt工程/流程设计 |

### 能力转型路径

```
Stage 1: 学会用AI
├── 熟悉AI工具能力边界
├── 掌握基础Prompt技巧
└── 建立AI协作习惯

Stage 2: 学会管AI
├── 设计多Agent协作架构
├── 建立质量关卡机制
└── 优化Session管理策略

Stage 3: 学会设计AI协作流程
├── 沉淀可复用模板
├── 建立团队规范
└── 培养他人AI协作能力
```

最可怕的不是AI抢你饭碗，是你还在用旧的技能树打新的副本。
