## 课程设计理念

### 核心认知转变
```
传统思维：我要学会写代码
AI时代思维：我要学会管理AI写代码

传统程序员    →    AI时代管理者
追求一次写对       追求快速迭代
亲手实现功能       描述需求、验收产出
技术能力是核心     管理能力是核心
```

### 课程结构逻辑
```
认知重塑（2节）→ 需求澄清（2节）→ 设计决策（2节）→ 
代码生成（4节）→ 测试验证（3节）→ 部署运维（2节）→ 综合实战（1节）

             上游战略层（高价值区）
                   ↑
    需求澄清 → 设计决策 → 代码生成 → 测试验证 → 部署运维
                   ↓
             人主导 ←───────→ AI主导
```

## 第1节：开篇 - 为什么管好AI比写好代码更重要

### 课程标题
**《震惊！花了一万美金学AI编程，最后得出"要学Haskell"的结论？》**

### 核心知识点（5个）
1. **Vibe Coding的真相**：不看代码、Accept All、能跑就行？听起来很爽，但需要的不是随意，是管理能力
2. **两种思维，两种命运**：传统程序员 vs AI时代管理者的能力模型对比
3. **最大瓶颈：上下文切换**：三个agent session同时跑，犹豫的每一分钟整个系统都在空转
4. **一个反直觉的做法**：session跑久了越陷越深？关掉它，开个新的（上下文污染原理）
5. **新能力模型**：品味、判断力、系统思维、表达力——AI时代你的核心竞争力

### 实操案例/演示内容
**案例1：一万美金买教训复盘**
- 场景：让agent为Kotlin项目集成GCP Transcoding服务，只给一句话描述
- 结果：agent自己判断用Ktor-client手动实现RESTful接口，被骂"蠢"
- 演示：如果当时多给六个字"封装现有Java SDK"，结果会怎样？

**案例2：上下文污染演示**
- 演示：同一个需求，在"干净session"和"长上下文session"中分别执行
- 对比：长上下文模型的性能断崖式下跌（有效容量只有标称的60-70%）

### 课后作业/练习
1. **认知自检**：对照"传统程序员 vs AI时代管理者"表格，给自己打分，找出最需要提升的3个能力项
2. **工具准备**：注册Claude Code或Cursor，完成基础配置
3. **心态转变**：尝试用"管理者思维"而非"执行者思维"描述你最近的一个需求

## 第2节：观念重塑 - 从包工头到建筑师

### 课程标题
**《还在当包工头？AI时代程序员的核心竞争力彻底变了》**

### 核心知识点（4个）
1. **许愿 vs 工程**：Prompt → Accept → Run → Pray（Vibe Coding）vs Plan → Direct → Review → Test（Agentic Engineering）
2. **土木工程的启示**：盖楼有甲方、设计院、施工队、监理四方制约，为什么你让同一个agent干所有事？
3. **AI参与度倒金字塔**：代码生成70% → 测试60% → 部署55% → 设计30% → 需求10%
4. **人的核心价值重新定位**：AI越擅长执行，人越擅长决策

### 实操案例/演示内容
**案例：Chrome团队Addy Osmani的核心洞察**
> "The single biggest differentiator between agentic engineering and vibe coding is **testing**."

**演示：两种模式对比**
| 维度 | Vibe Coding | Agentic Engineering |
|-----|-------------|---------------------|
| 工作方式 | 扔需求，等奇迹 | 拆任务，建流程 |
| Agent使用 | 单agent包办一切 | 多角色分工协作 |
| 质量保障 | 代码风格review | 独立测试验证 |
| 上下文管理 | 无限累积 | Session主动管理 |

### 课后作业/练习
1. **模式识别**：回顾你过去一周的AI协作经历，判断自己更像"Vibe Coding"还是"Agentic Engineering"
2. **流程设计**：选择一个你熟悉的功能模块，尝试设计"多角色分工"的协作流程（至少包含coder、tester、reviewer三个角色）
3. **效率测算**：估算你的"上下文切换效率"——AI跑完只要几分钟，你判断结果需要多久？

## 第3节：需求澄清（上）- AI时代最被低估的核心能力

### 课程标题
**《需求不清是项目失败的首要原因：AI只能放大错误，不能纠正方向》**

### 核心知识点（5个）
1. **需求澄清为什么最重要**：需求质量直接决定AI生成代码的质量，错误的需求会被AI放大传播
2. **AI的局限性**：AI可以快速实现需求，但无法判断需求的合理性
3. **传统模式 vs AI增强模式**：需求澄清周期从3-7天缩短到0.5-2天
4. **快速需求评估法**：15分钟判断需求健康度（0-10分评分）
5. **AI参与度的动态变化**：业务价值判断(5%) ← 人主导 → 场景生成(85%) ← AI辅助

### 实操案例/演示内容
**案例：秒杀功能需求澄清实战**

初始需求：
> "我们需要一个秒杀功能，用户可以在特定时间抢购商品。"

演示5W2H深度追问：
- **WHAT**：秒杀的具体规则是什么？（限购数量、库存扣减规则、超卖处理）
- **WHY**：为什么要做秒杀？（清库存？拉新？品牌宣传？）
- **WHO**：目标用户是谁？（新老用户？会员用户？）
- **WHEN**：秒杀时间段？（整点秒杀？限时秒杀？）
- **WHERE**：在哪个渠道？（App？小程序？H5？）
- **HOW**：技术方案是什么？（前端倒计时？后端库存扣减？）
- **HOW MUCH**：预计投入多少？（预算？服务器资源？）

**AI辅助演示**：用AI生成结构化问题清单，实时整理需求

### 课后作业/练习
1. **需求评估练习**：拿到一个模糊需求，15分钟内完成快速评估，打分并列出关键缺失信息
2. **5W2H实战**：选择一个你工作中遇到的真实需求，用5W2H框架完整澄清
3. **AI协作练习**：尝试用AI生成追问清单，并记录人机协作的效果

## 第5节：设计决策（上）- 架构思维是AI时代的护城河

### 课程标题
**《设计决策决定系统上限：AI可以提供方案，但选择需要你的判断》**

### 核心知识点（5个）
1. **设计决策的本质**：问题理解 × 方案探索 × 权衡分析 × 决策记录
2. **设计决策的成本**：错误的决策后期修复成本是前期的10-100倍
3. **问题定义框架**：当前状态、期望状态、差距分析、影响范围、紧迫性评估
4. **约束分析三维度**：业务约束、技术约束、组织约束
5. **AI在设计决策中的角色**：提供方案 vs 判断适用性

### 实操案例/演示内容
**案例：秒杀功能架构设计 - 问题定义**

```markdown
## 问题陈述

### 当前状态
- 促销活动期间系统压力大，经常崩溃
- 用户抢购体验差，投诉多
- 库存管理混乱，出现超卖

### 期望状态
- 支持高并发秒杀活动
- 用户体验流畅
- 库存准确，不超卖

### 约束条件
- 时间：2个月内上线
- 成本：预算有限，不能大规模扩容
- 团队：5人，熟悉Java和Redis
- 技术栈：Spring Cloud + MySQL + Redis
```

**AI辅助演示**：
```

{问题定义}

{约束条件}

请帮我生成至少3个技术方案，每个方案要包含：
1. 方案概述（核心思路）
2. 技术架构（关键组件）

## 第6节：设计决策（下）- ADR驱动设计与演进规划

### 课程标题
**《架构决策记录ADR：让每一次选择都有据可查》**

### 核心知识点（4个）
1. **方案对比矩阵**：多维度加权评分，让选择有数据支撑
2. **权衡分析模板**：性能vs成本、灵活性vs复杂度、短期收益vs长期价值
3. **ADR（Architecture Decision Record）**：背景、决策、后果、验证的完整记录
4. **演进规划**：分阶段实施、技术债务管理、演进触发条件

### 实操案例/演示内容
**案例：秒杀方案对比与决策**

方案对比矩阵：
| 维度 | 权重 | 方案A：数据库乐观锁 | 方案B：Redis预扣库存 | 方案C：消息队列削峰 |
|-----|------|-------------------|---------------------|-------------------|
| 性能 | 25% | 低 | 高 | 高 |
| 复杂度 | 20% | 低 | 中 | 高 |
| 成本 | 20% | 低 | 中 | 高 |
| 风险 | 20% | 高 | 中 | 低 |
| 团队能力匹配 | 15% | 高 | 高 | 中 |
| **加权总分** | 100% | 60 | **85** | 75 |

**ADR文档示例**：
```markdown
# ADR-001: 使用Redis预扣库存方案

## 状态
已接受

## 背景
促销活动期间系统压力大，需要支持高并发秒杀

## 决策
采用Redis预扣库存方案实现秒杀功能

### 决策理由
1. 性能满足要求（支持1万QPS）
2. 复杂度可控，团队能力匹配
3. 成本在预算范围内
4. 风险可通过异步同步机制控制

## 后果
### 正面影响
- 支持高并发场景
- 减轻数据库压力

### 负面影响
- 需要保证Redis和数据库一致性
- 增加了系统复杂度

### 风险与缓解
| 风险 | 缓解措施 |
|-----|---------|
| Redis和MySQL数据不一致 | 异步同步 + 对账机制 |
| Redis单点故障 | Redis集群 + 哨兵模式 |
```

### 课后作业/练习
1. **方案对比练习**：为第5节作业中生成的方案建立对比矩阵，做出选择
2. **ADR编写**：为一个重要的技术决策编写ADR文档
3. **演进规划**：为你的项目制定分阶段实施计划

## 第7节：代码生成（上）- Prompt工程：代码质量的决定因素

### 课程标题
**《Prompt质量决定代码质量：一次写对不是梦》**

### 核心知识点（5个）
1. **新的编程范式**：需求 → 设计 → Prompt → AI生成 → 人工审查 → 测试
2. **AI的能力边界**：AI是"高产但容易犯错的初级搭档"
3. **Prompt设计框架**：角色 + 任务 + 背景 + 要求 + 约束 + 示例 + 输出格式
4. **上下文完整性原则**：业务上下文 + 技术上下文 + 质量上下文
5. **渐进式生成原则**：复杂任务分步骤完成

### 实操案例/演示内容
**对比演示：差的Prompt vs 好的Prompt**

❌ 差的Prompt：
```
"写一个登录功能"
```

✅ 好的Prompt：
```markdown
## 角色
你是一个Spring Boot开发专家，擅长实现业务逻辑。

## 任务
请帮我实现用户登录功能。

## 背景
我正在开发一个电商系统的用户登录功能。
技术栈：Spring Boot + MyBatis + MySQL
业务场景：用户通过手机号和密码登录

## 要求
1. 支持手机号+密码登录
2. 密码需要加密存储
3. 登录成功返回JWT Token
4. 记录登录日志
5. 支持登录失败次数限制

## 约束
- 所有public方法都要有JavaDoc注释
- 所有参数都要进行非空校验
- 遵循阿里巴巴Java编码规范

## 示例
请参考以下代码风格：
```java
@Service
public class UserService {
    private final UserRepository userRepository;
    
    public UserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }
}
```

请先设计User实体类和LoginRequest DTO类。
```

## 第8节：代码生成（中）- AI代码生成与迭代优化

**《AI生成的代码有问题？迭代优化的艺术》**

2. **分步生成策略**：接口定义 → 核心逻辑 → 辅助方法 → 测试代码 → 文档注释
3. **迭代优化技巧**：明确指出问题、要求修改、保持功能不变
4. **代码生成检查清单**：结构检查、逻辑检查、规范检查
