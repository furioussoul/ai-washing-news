# 第15课：上下文管理——让你的AI团队更聪明

> 课程模块：进阶能力 | 预计时长：60分钟

---

## 一、课程导入

### 一个现象

新项目惊艳，老项目拉垮。上下文越长，AI越蠢。

这个现象有研究支撑。长上下文注意力不均匀，到某个阈值后性能断崖式下跌，有效容量通常只有标称最大值的60-70%。

**但知道问题和解决问题是两回事。**

---

## 二、核心内容

### 2.1 上下文污染的表现

#### 表现1：重复错误

AI在同一个问题上反复犯错，每次修复都引入新问题。

#### 表现2：记忆混乱

AI把之前的需求和当前的需求混淆，输出不相关的内容。

#### 表现3：质量下降

同样的Prompt，之前输出很好，现在输出质量明显下降。

---

### 2.2 上下文管理策略

#### 策略1：Session拆分

**原则**：一个Session只做一件事，做完就关闭。

```
❌ 错误做法：
一个Session里：设计数据库 → 写API → 写前端 → 修bug → 加功能

✅ 正确做法：
Session 1：设计数据库
Session 2：写API
Session 3：写前端
Session 4：修bug（如果发现bug）
Session 5：加新功能
```

#### 策略2：上下文沉淀

关闭Session前，让AI总结关键信息：

```
Prompt：
"请总结我们刚才讨论的关键决策和设计，
整理成文档格式，方便下次继续。"

AI输出：
## 项目设计文档

### 技术栈
- 前端：Next.js + Tailwind CSS
- 后端：Node.js + Express
- 数据库：PostgreSQL

### 数据库设计
- users表：id, email, password, created_at
- posts表：id, title, content, user_id, created_at

### 关键决策
1. 使用JWT进行认证
2. 密码使用bcrypt加密
3. API采用RESTful设计
```

#### 策略3：定期清理

**信号**：当出现以下情况时，考虑清理上下文

- AI开始重复之前的错误
- AI给出明显矛盾的答案
- AI说"根据之前的讨论"但内容不对
- 响应速度明显变慢

**做法**：
1. 让AI总结当前进度
2. 关闭Session
3. 开新Session，传入总结文档
4. 继续工作

---

### 2.3 长文本处理技巧

#### 技巧1：分块处理

长文档不要一次性给AI，分块处理：

```
❌ 错误做法：
"请分析这份100页的文档"

✅ 正确做法：
"这是文档的第1-10页，请总结关键信息"
（等AI回复后）
"这是第11-20页，请继续总结"
```

#### 技巧2：索引引用

给AI提供索引，而不是完整内容：

```
"我们有一个用户表，结构如下：
- users: id, name, email, created_at

请基于这个表结构，设计查询逻辑"
```

#### 技巧3：外部存储

大型上下文存储在外部，AI需要时引用：

```
"请参考项目文档：https://example.com/docs
请实现用户登录功能"
```

### 2.4 AI TEAM中的上下文管理

#### 跨任务上下文共享

```
Board: 个人博客项目

Task 1: 设计数据库
├── 输出：数据库设计文档
└── 保存到：项目知识库

Task 2: 实现API
├── 输入：数据库设计文档
├── 输出：API实现
└── 保存到：项目知识库

Task 3: 开发前端
├── 输入：API文档
└── 输出：前端实现
```

#### Agent间的上下文传递

```
Architect Agent完成任务后：
├── 生成：架构设计文档
├── 保存到：共享上下文
└── 通知：Developer Agent可以开始

Developer Agent开始任务时：
├── 读取：架构设计文档
├── 读取：相关技术规范
└── 基于上下文开发
```

---

### 2.5 一个反直觉的做法

一个session跑久了，每次修复都引入新问题，越陷越深。

这时候应该：**关掉这个session，开个新的。**

为什么？上下文被污染了。

拖着满是错误尝试的长上下文，模型会变笨。让当前Agent把经验沉淀到文档，然后clear session，换全新的继续。

就像团队里一个人在错误方向钻太久，你需要替他做那个他自己做不了的决定：**停下来。**

## 四、小结与思考题

### 本节要点

1. 上下文污染会导致AI性能下降
2. Session拆分、上下文沉淀、定期清理是三大策略
3. 长文本要分块处理，大型上下文要外部存储
4. 学会kill session，比学会写代码更重要

### 思考题

1. 你之前有没有遇到过上下文污染的情况？
2. 如果管理一个长期项目，你会怎么设计上下文管理策略？
