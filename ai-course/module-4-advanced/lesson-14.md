# 第14课：如何验收AI的产出——质量把关技巧

> 课程模块：进阶能力 | 预计时长：60分钟

---

## 一、课程导入

### 一个陷阱

AI生成了代码，测试全绿，上线后才发现字段名写错了。

困惑："测试也是AI写的，错的一致就是'对'的"。

**这就是验收的陷阱：让同一个AI既写代码又写测试。**

---

## 二、核心内容

### 2.1 验收的基本原则

#### 原则一：独立验证

让不同的Agent扮演不同角色：
- Coder：写代码
- Tester：写测试
- Reviewer：做审查

**每个角色独立，互相制约。**

#### 原则二：结果导向

不看代码怎么写的，看结果对不对：
- 功能是否符合需求？
- 边界情况是否处理？
- 性能是否达标？

#### 原则三：分层验收

| 层级 | 验收内容 | 验收者 |
|-----|---------|--------|
| 单元 | 函数逻辑正确 | Tester Agent |
| 集成 | 模块协作正常 | Tester Agent |
| 系统 | 完整功能可用 | 人工验收 |
| 验收 | 符合业务需求 | 人工验收 |

---

### 2.2 代码审查清单

#### 功能性审查

- [ ] 是否实现了需求中的所有功能？
- [ ] 边界情况是否处理？
- [ ] 错误处理是否完善？
- [ ] 输入验证是否充分？

#### 代码质量审查

- [ ] 代码是否可读？
- [ ] 是否有重复代码？
- [ ] 命名是否清晰？
- [ ] 注释是否必要且准确？

#### 性能审查

- [ ] 是否有明显的性能问题？
- [ ] 是否有不必要的循环或递归？
- [ ] 数据库查询是否优化？
- [ ] 内存使用是否合理？

#### 安全审查

- [ ] 是否有SQL注入风险？
- [ ] 是否有XSS风险？
- [ ] 敏感数据是否加密？
- [ ] 权限控制是否正确？

---

### 2.3 测试策略

#### 测试金字塔

```
        /\
       /  \     E2E测试（少量，关键路径）
      /____\
     /      \
    /        \   集成测试（中等数量）
   /__________\
  /            \
 /              \ 单元测试（大量，核心逻辑）
/________________\
```

#### 测试类型

| 类型 | 目的 | 工具 |
|-----|------|------|
| 单元测试 | 验证函数逻辑 | Jest, Mocha |
| 集成测试 | 验证模块协作 | Jest, React Testing Library |
| E2E测试 | 验证完整流程 | Playwright, Cypress |
| 性能测试 | 验证性能指标 | Lighthouse, k6 |
| 安全测试 | 发现安全漏洞 | OWASP ZAP, SonarQube |

---

### 2.4 验收流程

#### 步骤1：自动化测试

```
AI生成代码 → AI生成测试 → 运行测试 → 通过/失败
```

**注意**：测试代码要由不同的Agent生成。

#### 步骤2：代码审查

```
Reviewer Agent审查 → 提出问题 → Coder修复 → 重新审查
```

#### 步骤3：人工验收

```
部署到测试环境 → 人工测试 → 记录问题 → 反馈修复
```

#### 步骤4：验收标准核对

对照需求文档的验收标准，逐项检查：

```
需求：用户登录功能

验收标准核对：
- [x] 输入正确的邮箱密码，登录成功
- [x] 输入错误的密码，显示错误提示
- [x] 邮箱格式错误时，前端校验不通过
- [ ] 登录成功后，30分钟内免重新登录 ← 未实现
```

---

### 2.5 常见问题处理

#### 问题1：测试通过了，但功能不对

**原因**：测试没有覆盖正确的场景。

**解决**：
1. 检查测试用例是否正确
2. 补充缺失的测试场景
3. 人工验证功能

#### 问题2：AI反复修改，越改越错

**原因**：上下文污染，AI陷入局部最优。

**解决**：
1. 关闭当前session
2. 新开session，重新描述需求
3. 或者，人工介入修复

#### 问题3：代码能跑，但质量很差

**原因**：AI只关注功能，不关注质量。

**解决**：
1. 在需求中明确要求代码质量
2. 使用代码审查清单
3. 要求AI重构代码
