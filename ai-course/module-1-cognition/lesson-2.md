## 一、课程导入

### 去年有个词特别火

Vibe Coding。创始人说得很玄乎——不看代码，对着AI说话，Accept All，有错粘回去修，能跑就行。

很多人试完发现一个尴尬的事实：**这玩意儿需要的能力，和写代码完全是两回事。**

更尴尬的是，很多人以为自己在做"AI编程"，其实只是在"许愿"。

---

### 一个昂贵的教训

有人花了一万美金买token，试遍了Opus、Claude、Gemini等顶级模型，最后得出的结论是：AI coding不行，得回去学Rust和Haskell。

一万美金。这是50个月的Claude Max订阅费，或者一台顶配MacBook Pro。

投入这么多，最后的药方是"回归传统CS"？

**问题不在AI，在用法错了。**

## 二、核心内容

### 2.1 两种AI编程模式

Vibe Coding听起来很随意，实际上有它的价值——快速验证想法，降低入门门槛。

但很多人把它当成"AI编程的全部"，这就出问题了。

| Vibe Coding | Agentic Engineering |
|-------------|---------------------|
| Prompt → Accept → Run → Pray | Plan → Direct → Review → Test |
| 扔需求，等奇迹 | 拆任务，建流程 |
| 单Agent包办一切 | 多角色分工协作 |
| 代码风格review | 独立测试验证 |
| 上下文无限累积 | Session主动管理 |

**核心区别**：一个在许愿，一个在工程化管理。

---

### 2.2 Vibe Coding的三个典型误区

#### 误区一：一句话需求，期待读心术

让AI为Kotlin项目集成GCP服务，只给了一句话描述。AI通读文档后，决定用Ktor-client手动实现RESTful接口。

然后你抱怨AI"蠢"，说人类一眼就知道该封装Java SDK。

等等。你告诉新来的实习生"把这个功能做了"，他会怎么选技术路线？他当然会自己判断。如果你想让他封装SDK，你得明确说"封装现有Java SDK"。

**六个字，能省下一万美金。**

这不是AI能力不足，恰恰证明AI很听话：你说集成，它就集成；你说封装，它就封装。你不说，它只能猜。

**工程管理的第一课：需求不明确，神仙也救不了。**

---

#### 误区二：既当出题人，又当答题人

AI写了上千行getter/setter的测试用例，测试全绿，上线后才发现字段名写错了。

困惑："测试也是AI写的，错的一致就是'对'的"。

这个问题，软件工程界二十年前就解决了。叫TDD，叫测试先行，叫独立验证。

让同一个AI既写代码又写测试，相当于让考生自己出卷自己答，然后惊讶地发现他每次都满分。

**正确的做法**：让不同的Agent扮演不同角色——coder写代码，tester写测试，reviewer做审查。每个角色独立，互相制约。

Google Chrome团队的Addy Osmani说得很直白：

> *The single biggest differentiator between agentic engineering and vibe coding is testing.*

**测试。就这么简单。**

---

#### 误区三：新鲜感红利吃尽，怪AI变笨

新项目惊艳，老项目拉垮。上下文越长，AI越蠢。

这个现象有研究支撑。长上下文注意力不均匀，到某个阈值后性能断崖式下跌，有效容量通常只有标称最大值的60-70%。

但知道问题和解决问题是两回事。很多人写了context文档，但没有做session拆分、任务边界控制、上下文清理。

就像一个人连续工作72小时然后怪他犯错——你不会这样对待人类同事，为什么要这样对待AI？

**在我们社区有个说法：如果你触发了AI的自动压缩，说明你还停留在入门阶段。**

---

### 2.3 什么是真正的工程化

所有严肃从业者都指向同一个方向：**把AI当工程问题管理，而不是当魔法许愿机。**

工程化意味着：

1. **Plan（规划）**：拆任务、定优先级、设检查点
2. **Direct（指导）**：给清晰的需求、上下文、约束条件
3. **Review（审查）**：代码审查、架构审查、安全审查
4. **Test（测试）**：单元测试、集成测试、端到端测试

这不是什么新概念，是软件工程几十年积累下来的最佳实践。只是现在，执行者从人变成了AI。

**你从执行者变成了管理者。**

---

### 2.4 一个正确的例子

一个13个Agents、7个Skills的团队架构：

| 角色 | 职责 |
|-----|------|
| Architect | 把控设计，技术选型 |
| Developer | 写代码 |
| Tester | 写测试 |
| Reviewer | 做审查 |
| ... | ... |

6996行代码跑下来，外部review只发现两个问题，还都是手动干预的结果。

**不是靠运气，是靠流程。**

## 三、实操环节

### 练习1：诊断你的AI编程模式

回想你最近一次用AI写代码的经历，回答以下问题：

| 问题 | 你的回答 |
|-----|---------|
| 你给AI的需求描述有多详细？ | 一句话 / 一段话 / 一份文档 |
| AI生成的代码，你是怎么验证的？ | 跑一下看 / 读代码 / 写测试 |
| 出问题时，你是怎么处理的？ | 让AI改 / 自己改 / 重新描述需求 |
| 你的上下文管理方式是？ | 一直用同一个session / 定期清理 |

如果你的答案都在左边，你很可能在做Vibe Coding。

## 四、小结与思考题

### 本节要点

1. Vibe Coding不是AI编程的全部，只是一个起点
2. 三个典型误区：需求模糊、角色混淆、上下文失控
3. Agentic Engineering = Plan + Direct + Review + Test
4. 你从执行者变成了管理者

### 思考题

1. 你之前的AI编程经历，属于哪种模式？有哪些问题？
2. 如果要用Agentic Engineering的方式重新做一次，你会怎么改进？
