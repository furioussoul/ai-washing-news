# 第3课：从写代码到管理AI团队

> 课程模块：认知升级 | 预计时长：60分钟

## 一、课程导入

### 一个反直觉的转变

你以为在编程，其实在带团队。

去年Vibe Coding火的时候，很多人以为AI编程就是"对着AI说话"。试完之后发现：这需要的能力和写代码完全是两回事。

**不看代码看什么？**

传统编程默认代码应该一次写对，核心是对过程的控制感。Vibe Coding把这个假设撕了。你不看代码，当然不可能对过程有信心。

那怎么办？看结果。

---

### 最大瓶颈：上下文切换

三个Agent session同时跑，你是唯一的人类。

Agent跑完只要几分钟，你判断结果需要多久？犹豫的每一分钟，整个系统都在空转。

好的leader三分钟给结论，差的leader犹豫一天。

**你跟不上，就变成瓶颈。**

## 二、核心内容

### 2.1 管理者的日常

想象一下，你是一个技术团队的leader：

- 早上，前端工程师说某个功能实现不了
- 中午，后端工程师提交了代码等你review
- 下午，测试工程师发现了一个严重bug
- 晚上，产品经理催进度

你的工作是：
1. 判断前端的问题是真做不了，还是需要换种思路
2. 快速review代码，给出反馈
3. 评估bug的严重程度，决定修复优先级
4. 和产品经理沟通，调整预期

**注意：你没有写一行代码。**

---

### 2.2 AI团队 vs 人类团队

现在，把这个场景换成AI：

| 场景 | 人类团队 | AI团队 |
|-----|---------|--------|
| 前端说做不了 | 开会讨论，换方案 | 开个新session，重新描述需求 |
| 代码review | 逐行看代码 | 看结果对不对，不看过程 |
| 发现bug | 分配修复任务 | 让AI自己修，或重新生成 |
| 进度压力 | 加班赶工 | 并行开多个session |

**核心区别**：人类团队的管理是"协调人"，AI团队的管理是"设计机制"。

---

### 2.3 一个反直觉的做法

一个session跑久了，每次修复都引入新问题，越陷越深。

这时候应该：**关掉这个session，开个新的。**

为什么？上下文被污染了。

拖着满是错误尝试的长上下文，模型会变笨。让当前Agent把经验沉淀到文档，然后clear session，换全新的继续。

就像团队里一个人在错误方向钻太久，你需要替他做那个他自己做不了的决定：**停下来。**

---

### 2.4 最贵的不是订阅费

Claude Code Max一个月两三百美金。但你算一笔账：

弱模型出现幻觉，你花半小时排查纠偏。贵的那个一次就对了。

**你的时间才是整个系统里最贵的资源。**

---

### 2.5 新能力模型

放弃过程掌控不代表放弃质量。质量保障方式变了：从过程审查变成机制设计。

**你需要的新能力：**

| 能力 | 说明 | 培养方法 |
|-----|------|---------|
| 品味 | 知道什么是好结果 | 多看优秀案例，建立审美 |
| 判断力 | 什么时候质疑，什么时候kill session | 快速试错，积累经验 |
| 系统思维 | 怎么设计机制让AI团队自运转 | 学习系统工程，设计流程 |
| 表达力 | 需求描述的清晰度决定产出质量 | 练习写作，结构化表达 |

### 练习2：决策速度训练

以下场景，你会怎么做？限时3分钟决策：

**场景A**：AI生成的代码能跑，但你看不懂
- [ ] 让AI解释清楚
- [ ] 直接接受，能跑就行
- [ ] 要求AI重写，直到你能看懂

**场景B**：一个session跑了10轮，bug越修越多
- [ ] 继续让AI修
- [ ] 自己上手改
- [ ] 关掉session，重新开始

**场景C**：两个Agent给出了不同的方案
- [ ] 选第一个
- [ ] 选第二个
- [ ] 让两个Agent讨论，你旁观
- [ ] 综合两个方案，重新描述需求

## 四、小结与思考题

### 本节要点

1. AI编程的本质是管理，不是写代码
2. 核心能力：品味、判断力、系统思维、表达力
3. 最贵的资源是你的时间，不是订阅费
4. 学会kill session，比学会写代码更重要

### 思考题

1. 如果你现在管理一个3人AI团队，你最担心什么问题？
2. 对照四种新能力，你打算如何提升最弱的那一项？
