## 课程定位与核心价值

### 核心价值主张

**一句话定位**：用AI团队（而非AI工具）开发你的第一个App

这不是教你怎么用Cursor、Copilot这些工具，而是教你**管理一个AI团队**——像项目经理一样拆任务、像架构师一样做决策、像测试工程师一样验收质量。

### 核心理念来源

> **"管好AI比写好代码更重要"**
> 
> 你不再写代码，你是让代码被正确写出来的人。管的这个团队，恰好由AI组成。
> 
> —— Vibe Coding 核心洞察

### AI时代的能力转变

| 传统程序员 | AI时代管理者 |
|-----------|-------------|
| 追求一次写对，盯着每行代码 | 追求快速迭代，盯着最终结果 |
| 亲手实现功能，怕AI写错代码 | 描述需求、验收产出，怕自己描述不清 |
| 抢过来自己写，瓶颈是手速 | 忍住让别人写，瓶颈是判断速度 |
| 技术能力是核心 | 管理能力是核心 |

### AI参与度金字塔

```
┌────────────────────────────────────────────────────────┐
│  代码生成 (70%)  ████████████████████████████████████  │  AI主导
│  测试验证 (60%)  ██████████████████████████████        │
│  部署运维 (55%)  ████████████████████████████          │
│  设计决策 (30%)  ██████████████████                    │
│  需求澄清 (10%)  ██████                                │  人主导
└────────────────────────────────────────────────────────┘
```

**关键洞察**：
- 上游越重要：需求和设计阶段的错误会被AI放大传播
- AI越擅长执行：代码生成、测试数据生成等执行层工作AI参与度最高
- 人越擅长决策：业务理解、架构权衡、应急响应等需要人主导

### 第1课：AI时代的开发者——不做程序员，做产品工程师

#### 课程标题

**AI时代的开发者：不做程序员，做产品工程师**

#### 学习目标

1. ✅ 理解AI时代开发者角色的本质变化
2. ✅ 建立"管理AI"而非"使用AI"的正确心态
3. ✅ 了解AI编程的能力边界和常见误区
4. ✅ 明确本课程的学习路径和预期成果

#### 核心知识点

##### 1.1 AI时代的能力模型重构

**传统能力金字塔 vs AI时代能力金字塔**：

```
传统模式：                    AI时代模式：
     算法                          业务理解
       ↑                              ↑
    架构设计                        架构决策
       ↑                              ↑
   编码能力                        AI协作能力
       ↑                              ↑
  语言语法                        Prompt工程
```

**核心转变**：
- 从"我能写什么代码"到"我能让AI写出什么代码"
- 从"技术深度"到"产品广度+技术判断力"
- 从"执行者"到"管理者"

##### 1.2 Vibe Coding vs Agentic Engineering

| Vibe Coding（许愿式） | Agentic Engineering（工程化） |
|---------------------|---------------------------|
| Prompt → Accept → Run → Pray | Plan → Direct → Review → Test |
| 扔需求，等奇迹 | 拆任务，建流程 |
| 单Agent包办一切 | 多角色分工协作 |
| 代码风格Review | 独立测试验证 |
| 上下文无限累积 | Session主动管理 |

**关键洞察**：
> AI编程不是许愿，而是工程化管理。花了一万美金买token却得出"要学Haskell"的结论——这不是AI的错，是用法错了。

##### 1.3 三个典型误区

**误区一：一句话需求，期待读心术**
- 问题：给AI一句话描述，期待AI自己理解所有细节
- 后果：AI按自己的理解做，结果与预期相差甚远
- 正确做法：提供充分的需求上下文

**误区二：既当出题人，又当答题人**
- 问题：让同一个AI既写代码又写测试
- 后果：测试全绿，但业务逻辑错误
- 正确做法：多角色分工，互相制约

**误区三：上下文无限累积，怪AI变笨**
- 问题：一个session跑到底，不主动管理上下文
- 后果：AI越跑越蠢，错误越来越多
- 正确做法：主动拆分session，清理上下文

##### 1.4 最大瓶颈：上下文切换

三个AI session同时跑，你是唯一的人类：
- Agent跑完只要几分钟
- 你判断结果需要多久？
- 犹豫的每一分钟，整个系统都在空转

**好的Leader**：三分钟给结论  
**差的Leader**：犹豫一天

你跟不上，就变成瓶颈。

#### 核心知识点

##### 2.1 AI编程工具全景

| 工具类型 | 代表工具 | 核心特点 | 适用场景 |
|---------|---------|---------|---------|
| **IDE集成** | Cursor, GitHub Copilot, Windsurf | 实时代码补全，上下文感知 | 日常编码、代码补全 |
| **对话式** | ChatGPT, Claude, Gemini | 多轮对话，擅长解释和分析 | 方案设计、问题排查 |
| **Agent式** | Claude Code, Devin, OpenHands | 自主执行，自动调试 | 复杂任务、端到端开发 |

**工具选择建议**：
- 新手入门：Cursor + ChatGPT组合
- 进阶实践：Cursor + Claude Code
- 独立项目：Claude Code为主，配合其他工具

##### 2.2 AI团队的13角色架构

借鉴"13个agents配7个skills"的理念：

```
┌─────────────────────────────────────────────────────┐
│                    你（人类管理者）                   │
│                   决策 · 验收 · 协调                  │
└─────────────────────────────────────────────────────┘
                          │
        ┌─────────────────┼─────────────────┐
        │                 │                 │
   ┌────▼────┐      ┌────▼────┐      ┌────▼────┐
   │ 需求澄清 │      │ 设计决策 │      │ 代码生成 │
   │ Agent   │      │ Agent   │      │ Agent   │
   └─────────┘      └─────────┘      └─────────┘
        │                 │                 │
   ┌────▼────┐      ┌────▼────┐      ┌────▼────┐
   │ 代码审查 │      │ 测试验证 │      │ 部署运维 │
   │ Agent   │      │ Agent   │      │ Agent   │
   └─────────┘      └─────────┘      └─────────┘
```

**角色分工**：

| 角色 | 职责 | 你需要做什么 |
|------|------|-------------|
| **需求澄清Agent** | 帮你梳理需求、发现遗漏 | 回答问题、确认优先级 |
| **设计决策Agent** | 提供技术方案、权衡利弊 | 做出决策、确认方案 |
| **代码生成Agent** | 编写代码实现功能 | 审查代码、提出修改要求 |
| **代码审查Agent** | 审查代码质量、发现问题 | 确认问题严重性、决定修复策略 |
| **测试验证Agent** | 编写测试用例、执行测试 | 验收测试结果、确认覆盖率 |
| **部署运维Agent** | 处理部署、监控告警 | 确认部署策略、处理告警 |

##### 2.3 Session管理的艺术

**核心原则**：上下文被污染时，果断开新session

**触发新Session的信号**：
1. 同一个问题修复超过3次仍有新问题
2. AI开始"遗忘"之前的约定
3. 生成的代码越来越不符合预期
4. 你感觉"这个session不太对"

**Session切换流程**：
```
1. 让当前Agent总结已完成的工作和发现的问题
2. 将经验沉淀到文档（可能是claude.md或agents.md）
3. Clear Session，开一个新的
4. 在新Session中提供背景文档，继续工作
```

##### 2.4 最贵的不是订阅费

Claude Code Max一个月两三百美金。但你算一笔账：
- 弱模型出现幻觉，你花半小时排查纠偏
- 强模型一次就对了
- 你的时间才是整个系统里最贵的资源

**ROI计算**：
- 你的一小时值多少钱？
- 如果节省10小时/周，工具成本是否值得？

### 第3课：需求澄清——别让AI猜你想做什么

#### 课程标题

**需求澄清：别让AI猜你想做什么**

#### 学习目标

1. ✅ 理解需求质量对AI编程效果的决定性影响
2. ✅ 掌握5W2H深度追问法
3. ✅ 学会识别需求中的边界条件和异常场景
4. ✅ 能够输出AI友好的需求文档

#### 核心知识点

##### 3.1 需求质量 = AI产出质量

**关键公式**：
```
AI代码质量 = 需求清晰度 × Prompt质量 × 审查严格度
```

**需求澄清的参与度**：
- AI参与度：**10%**
- 人的核心价值：业务洞察、利益平衡、隐性需求识别

**核心原则**：
> 需求不明确，神仙也救不了。你告诉AI"把这个功能做了"，它只能猜。你说什么，它做什么；你不说，它不会做。

##### 3.2 5W2H深度追问法

```
├── What: 做什么？核心功能是什么？
│   └── 追问：这个功能的输入输出是什么？
│
├── Why: 为什么做？解决什么问题？
│   └── 追问：不解决这个问题会怎样？
│
├── Who: 谁来用？涉及哪些角色？
│   └── 追问：不同角色的权限有什么区别？
│
├── When: 什么时候用？有时间约束吗？
│   └── 追问：超时怎么处理？
│
├── Where: 在哪用？什么场景？
│   └── 追问：不同场景的行为有区别吗？
│
├── How: 怎么做？流程是什么？
│   └── 追问：每一步失败了怎么办？
│
└── How much: 做多少？性能要求？
    └── 追问：数据量大了怎么办？
```

##### 3.3 边界条件穷举

**AI擅长**：正常流程  
**人必须做**：边界条件、异常场景

**常见边界条件类型**：

| 类型 | 示例 |
|------|------|
| **数值边界** | 年龄0岁、150岁、负数、非数字 |
| **字符串边界** | 空字符串、超长字符串、特殊字符 |
| **时间边界** | 闰年、跨时区、过期时间 |
| **状态边界** | 未登录、权限不足、数据不存在 |
| **并发边界** | 同时操作、重复提交 |

##### 3.4 AI友好型需求文档

**核心要素**：
1. **背景说明**：为什么要做这个功能
2. **功能描述**：具体要做什么
3. **输入输出**：每个接口的输入输出定义
4. **边界条件**：所有异常场景的处理方式
5. **约束条件**：技术约束、时间约束、资源约束
6. **验收标准**：如何判断功能完成

#### 实操案例

**案例：待办事项App的"添加任务"功能需求澄清**

初始需求（一句话）：

##### 4.2 MVP功能优先级排序

| **Should have** | 重要但可以后续迭代 | 截止日期提醒、标签分类 |
| **Could have** | 有更好，没有也可以 | 语音添加、主题切换 |
| **Won't have** | 本版本不做 | 团队协作、任务分享 |

#### 核心知识点

##### 5.1 技术选型决策

**技术选型考虑因素**：

```
├── 项目因素
│   ├── 项目规模：小型/中型/大型
│   ├── 团队背景：熟悉什么技术栈
│   └── 时间要求：快速上线 vs 长期维护
│
├── 技术因素
│   ├── 学习曲线：是否容易上手
│   ├── 生态成熟度：是否有足够文档和社区
│   └── AI友好度：AI是否擅长这个技术栈
│
└── 业务因素
    ├── 性能要求：高并发 vs 普通应用
    ├── 扩展需求：是否需要频繁迭代
    └── 特殊需求：如实时通信、AI能力等
```

**新手推荐技术栈**：

| 场景 | 推荐方案 | 理由 |
|------|---------|------|
| **纯前端展示** | React + Vercel | 部署简单，免费额度够用 |
| **有数据存储** | React + Node.js + SQLite | 全栈JS，学习成本低 |
| **快速MVP** | Next.js + Supabase | 一站式解决方案 |
| **移动端优先** | React Native + Expo | 一套代码两端运行 |

##### 5.2 架构决策记录（ADR）

**ADR模板**：
```markdown
# ADR-001: 使用SQLite作为数据库

## 背景
- 这是个人使用的待办App
- 预期数据量：每个用户最多1000条任务
- 需要支持离线使用

## 决策
使用SQLite作为数据库

## 理由
1. 轻量级，无需独立数据库服务
2. 支持离线使用
3. 数据迁移简单（就是一个文件）
4. 对于个人使用的App，性能足够

## 后果
- 不能支持大规模用户（但这是个人App，不是问题）
- 没有复杂的数据库特性（但这个场景不需要）
```

##### 5.3 数据模型设计

**核心原则**：
1. 从业务实体出发
2. 明确实体之间的关系
3. 考虑查询效率

**示例：待办事项App数据模型**

```
┌─────────────┐       ┌─────────────┐
│   User      │       │    Task     │
├─────────────┤       ├─────────────┤
│ id          │───┐   │ id          │
│ email       │   │   │ user_id     │───┐
│ password    │   │   │ title       │   │
│ created_at  │   │   │ description │   │
└─────────────┘   │   │ due_date    │   │
                  │   │ priority    │   │
                  │   │ status      │   │
                  │   │ created_at  │   │
                  │   └─────────────┘   │
                  │                     │
                  └─────────────────────┘
                   一个用户有多个任务
```

**ER图转SQL**：
```sql
CREATE TABLE users (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    email TEXT UNIQUE NOT NULL,
    password_hash TEXT NOT NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE tasks (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id INTEGER NOT NULL,
    title TEXT NOT NULL,
    description TEXT,
    due_date DATE,
    priority TEXT DEFAULT 'medium',
    status TEXT DEFAULT 'pending',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id)
);
```
