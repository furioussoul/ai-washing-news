## 一、课程导入

### 一个常见的问题

你想开发一个App，脑子里有一堆想法：
- 用户登录
- 数据存储
- 界面设计
- 支付功能
- 消息通知
- ...

**从哪开始？**

很多人直接打开AI，说："帮我开发一个App。"

然后发现：AI不知道你要什么，你也不知道怎么描述。

**问题不在AI，在任务没有拆解。**

---

### 看板的价值

看板（Kanban）最初来自丰田的生产管理，核心是：
- 可视化工作流程
- 限制在制品数量
- 持续改进

用在AI编程上，就是：
- 把大任务拆成小任务
- 每个任务有明确的状态
- 一眼看清整个项目的进度

## 二、核心内容

### 2.1 任务拆解的艺术

#### 错误的拆解

```
❌ 开发一个App
❌ 实现用户系统
❌ 写代码
```

问题：太模糊，无法执行。

#### 正确的拆解

```
✅ 设计用户表结构（字段：id, username, email, password_hash）
✅ 实现用户注册API（输入：username, email, password；输出：user_id）
✅ 实现用户登录API（输入：email, password；输出：token）
✅ 编写注册/登录的单元测试
```

特点：具体、可验收、可分配。

---

### 2.2 任务拆解的MECE原则

MECE = Mutually Exclusive, Collectively Exhaustive
（相互独立，完全穷尽）

#### 示例：用户系统

```
用户系统
├── 用户注册
│   ├── 前端：注册表单
│   ├── 后端：注册API
│   └── 测试：注册流程测试
├── 用户登录
│   ├── 前端：登录表单
│   ├── 后端：登录API
│   └── 测试：登录流程测试
├── 密码管理
│   ├── 密码加密存储
│   ├── 密码重置
│   └── 测试：安全性测试
└── 会话管理
    ├── Token生成
    ├── Token验证
    └── Token刷新
```

每个子任务：
- 可以独立完成
- 有明确的验收标准
- 可以分配给不同的Agent

---

### 2.3 看板列的设计

#### 基础版（4列）

```
Backlog → In Progress → Review → Done
```

- **Backlog**：待办任务池
- **In Progress**：正在进行中
- **Review**：已完成，等待审核
- **Done**：已验收完成

#### 进阶版（6列）

```
Backlog → Ready → In Progress → Review → Testing → Done
```

- **Ready**：已准备好，可以开始
- **Testing**：代码审查通过，进入测试

#### 专业版（8列+）

```
Backlog → Analysis → Ready → In Progress → Code Review → Testing → Deploy → Done
```

根据项目复杂度选择。

---

### 2.4 任务卡片的内容

一个完整的Task应该包含：

```
┌─────────────────────────────────────┐
│ 标题：实现用户注册API               │
├─────────────────────────────────────┤
│ 描述：                              │
│ 实现用户注册功能，包括：            │
│ - 接收username, email, password     │
│ - 验证输入格式                      │
│ - 检查email是否已存在               │
│ - 密码加密存储                      │
│ - 返回user_id                       │
├─────────────────────────────────────┤
│ 分配给：Developer Agent             │
│ 优先级：High                        │
│ 截止日期：2024-03-15                │
├─────────────────────────────────────┤
│ 验收标准：                          │
│ □ API能正确接收参数                 │
│ □ 输入验证正常工作                  │
│ □ 重复email返回错误                 │
│ □ 密码加密存储                      │
│ □ 单元测试通过率100%                │
├─────────────────────────────────────┤
│ 关联文档：                          │
│ - 技术设计文档                      │
│ - API接口规范                       │
└─────────────────────────────────────┘
```

---

### 2.5 依赖关系管理

任务之间常有依赖：

```
设计数据库表结构 → 实现后端API → 编写单元测试
        ↓
   前端界面开发
```

在AI TEAM中，可以设置：
- **阻塞关系**：Task B 依赖于 Task A，A未完成时B不能开始
- **触发关系**：Task A 完成后，自动创建 Task B
- **关联关系**：Task A 和 Task B 有关联，但无强制依赖

## 四、小结与思考题

### 本节要点

1. 任务拆解是AI编程的前提，拆解不清，AI也帮不了你
2. MECE原则：相互独立，完全穷尽
3. 看板列设计：根据项目复杂度选择4-8列
4. 任务卡片要包含：描述、验收标准、依赖关系

### 思考题

1. 你之前用AI编程时，有没有遇到"不知道怎么说清楚"的情况？现在知道怎么解决了吗？
2. 如果要开发你想做的产品，第一步应该拆解出什么任务？
