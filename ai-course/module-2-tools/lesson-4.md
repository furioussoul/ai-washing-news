# 第4课：认识你的AI团队——AI TEAM产品介绍

> 课程模块：工具上手 | 预计时长：60分钟

---

## 一、课程导入

### 一个问题

你已经知道了：AI编程不是许愿，是工程化管理。

你也知道了：你需要管理一个AI团队，而不是用一个AI工具。

**但问题是：怎么管？**

用ChatGPT一个窗口来回切换？用Cursor写代码，用Claude写文档，用GitHub Copilot补全？

这就像让团队成员各自在家办公，没有会议室，没有看板，没有协作工具。

**你需要一个"AI团队的管理系统"。**

---

### 这就是AI TEAM

一个看板式的AI任务管理系统，让多Agent自动化协同完成复杂任务。

不是又一个AI工具，是AI团队的管理系统。

---

## 二、核心内容

### 2.1 为什么需要AI TEAM

#### 现状的问题

| 问题 | 场景 | 后果 |
|-----|------|------|
| 上下文碎片化 | 一个项目开10个ChatGPT窗口 | 信息分散，难以追溯 |
| 角色混乱 | 同一个AI既写代码又写测试 | 质量无法保证 |
| 进度不可见 | 不知道AI做到哪一步了 | 无法预估时间 |
| 协作困难 | 多个AI任务之间没有关联 | 效率低下 |

#### AI TEAM的解决方案

```
看板式任务管理 + 多Agent协同 + 自动化工作流 = AI TEAM
```

---

### 2.2 AI TEAM的核心概念

#### 概念一：Board（看板）

一个Board对应一个项目。就像Jira里的Project，Trello里的Board。

看板上的每一列代表任务的状态：
- Backlog（待办）
- In Progress（进行中）
- Review（待审核）
- Done（已完成）

#### 概念二：Task（任务）

每个Task代表一个具体的工作项。可以分配给特定的Agent，设置优先级、截止日期。

Task包含：
- 标题和描述
- 分配给哪个Agent
- 当前状态
- 关联的上下文（文档、代码、对话历史）

#### 概念三：Agent（智能体）

每个Agent是一个专门的AI角色：
- Developer：写代码
- Tester：写测试
- Reviewer：做代码审查
- Architect：做技术设计
- Writer：写文档

每个Agent有自己的技能集、上下文记忆、工作偏好。

#### 概念四：Workflow（工作流）

定义任务之间的依赖关系和自动化规则：

```
当 Task A 完成时 → 自动创建 Task B
当 Task 进入 Review 时 → 通知 Reviewer Agent
当 Task 超过截止日期 → 标记为紧急
```

---

### 2.3 AI TEAM vs 传统工具

| 维度 | 传统AI工具 | AI TEAM |
|-----|-----------|---------|
| 使用方式 | 单窗口对话 | 看板式多任务管理 |
| 角色管理 | 无 | 多Agent角色分工 |
| 上下文 | 单session | 跨任务上下文共享 |
| 进度追踪 | 无 | 可视化看板 |
| 协作 | 无 | 自动化工作流 |
| 适用场景 | 简单任务 | 复杂项目 |

---

### 2.4 一个真实的例子

**场景**：开发一个待办事项App

**传统方式**：
1. 打开ChatGPT，描述需求
2. 生成代码，复制到IDE
3. 发现bug，回到ChatGPT修复
4. 重复步骤2-3，直到能跑
5. 忘记之前的需求描述，重新开始

**AI TEAM方式**：
1. 创建Board："待办事项App"
2. 创建Task："设计数据库表结构" → 分配给Architect Agent
3. Architect完成后，自动创建Task："实现后端API" → 分配给Developer Agent
4. Developer完成后，自动创建Task："编写单元测试" → 分配给Tester Agent
5. 所有Task完成后，自动部署

**区别**：前者是"对话"，后者是"项目管理"。

## 四、小结与思考题

### 本节要点

1. AI编程需要管理系统，不是单个工具
2. AI TEAM的核心：Board + Task + Agent + Workflow
3. 从"对话"升级到"项目管理"
